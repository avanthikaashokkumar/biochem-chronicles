Why I Started The BioChem Chronicles

A short introduction to the purpose of this site: using student-built projects, clear writing, and honest documentation to make biotechnology and bioinformatics easier to approach.

Project overview

The BioChem Chronicles is a static portfolio and science communication website. It includes Arduino projects, environmental monitoring systems, health monitoring prototypes, assistive technology, and planned bioinformatics work.

Status note: This is an introduction to a student portfolio. It does not claim formal publication or professional research experience.

Problem being solved

Biotechnology and biochemistry can feel distant when they are explained only through dense textbooks or technical papers. I wanted a place where I could explain what I am learning through small, hands-on projects and clear notes.

Why I built it

I built this site because I am interested in the space between biology, devices, and data. As I grow as a Rutgers undergraduate, I want to document my work in a way that professors, mentors, and other students can understand quickly.

How it works

The site uses simple static files that can run on GitHub Pages. JavaScript stores project and article data, then renders cards, filters, 3D-style explainers, and article previews.

Step-by-step build process

  1. Gather the strongest project writeups and rewrite them in a clearer voice.
  2. Label each project honestly as completed, prototype, planned, or research note.
  3. Create a separate article page for every project instead of leaving cards as placeholders.
  4. Add local custom diagrams so the site does not depend on copyrighted images.
  5. Check accessibility, links, paths, and mobile layout before publishing.

Calibration or testing

I checked the site by reviewing links, image paths, alt text, nav behavior, and the files needed for GitHub Pages.

Challenges faced

  • Keeping the writing honest without making the projects sound small.
  • Avoiding generic science language that sounds polished but not personal.
  • Making the site feel complete without claiming lab research that has not happened yet.

What I learned

  • Good documentation is part of the project, not an afterthought.
  • A prototype can still be valuable when its limits are stated clearly.
  • A portfolio should make it easy for a mentor to see how a student thinks.

Future improvements

  • Add real build photos after the physical projects are photographed cleanly.
  • Add GitHub repositories for each project once the code is organized.
  • Add short data visualizations for projects that collect sensor readings.

Safety or accuracy note

This website is for learning and portfolio documentation. The project pages should not be treated as professional engineering, clinical, or laboratory validation.
